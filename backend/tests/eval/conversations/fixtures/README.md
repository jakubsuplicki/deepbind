# Conversation eval fixtures

Hand-authored multi-turn conversation fixtures used by the conversation-replay eval harness ([ADR 010](../../../../../docs/architecture/decisions/010-conversation-replay-eval-harness.md)). Each fixture is replayed under multiple `ContextStrategy` implementations (full-history, naive-truncate, retrieval-substitution, …) and scored mechanically against `expected_facts` plus, optionally, by the LLM-judge protocol.

These fixtures are the contract. They define what *correct* looks like at the scored turns; everything else (chat-model choice, judge model, strategy implementations) is replaceable around them.

## Schema (v1)

Each fixture is a JSON file:

```jsonc
{
  "id": "long-conv-tool-recall",          // stable id; embedded in baseline filenames
  "schema_version": 1,                     // bumped on breaking schema changes
  "description": "...",                    // why this fixture exists, what it stresses
  "tags": ["long-conv", "tool-recall"],    // for filter / report grouping
  "profile": "mechanical-process-engineer",// ADR 005 profile this fixture targets

  "preconditions": {
    "workspace_fixture": "reference_workspace",  // SQLite + Markdown bootstrap
    "tool_mocks": "<id>.tools.json"              // sibling file or null
  },

  "turns": [
    { "role": "user",              "content": "..." },
    { "role": "assistant_scripted","content": "...",
      "expected_tool_calls": [{ "name": "...", "args": { ... } }] },
    { "role": "tool_result",       "tool_name": "...", "content": "..." },
    { "role": "assistant_target",
      "expected_facts": [
        { "id": "fact_a", "match": "regex", "pattern": "..." },
        { "id": "fact_b", "match": "fuzzy", "text": "...", "min_score": 0.75 }
      ],
      "must_not_contain": [
        { "id": "no_x", "match": "regex", "pattern": "..." }
      ]
    }
  ]
}
```

### Roles

| Role | Source of content | Purpose |
|---|---|---|
| `user`              | scripted in fixture | drives the conversation forward |
| `assistant_scripted`| scripted in fixture | builds the conversation history; **not generated by the model**, **not scored** |
| `tool_result`       | scripted in fixture (via the mock layer) | recorded tool output for the prior `assistant_scripted` turn's tool call |
| `assistant_target`  | **generated by the model under test** | the turn the harness scores |

The runner replays scripted turns verbatim into the model's history, then on each `assistant_target` turn it invokes the chat model with the active `ContextStrategy` to produce a response. The response is scored against that turn's `expected_facts` and `must_not_contain`. A fixture may have multiple `assistant_target` turns.

### Fact matching

- **`regex`** — Python `re` pattern; case-insensitive flag is the author's responsibility (use `(?i)` inline). The fact "passes" if `re.search(pattern, response)` returns a match.
- **`fuzzy`** — token-level fuzzy match against `text`; the fact "passes" if normalized similarity ≥ `min_score`. Use for paraphrase-tolerant checks (e.g., "30-day MACE rate" should match even if reordered as "MACE rate at 30 days").

### Mechanical scoring rule

A turn **mechanically passes** if every `expected_facts` entry passes AND no `must_not_contain` entry matches. A mechanical fail short-circuits the turn — judge score is recorded but does not count toward the win-rate aggregation.

## Tool mock layer

When `preconditions.tool_mocks` is set, the runner loads the sibling `<id>.tools.json` file:

```jsonc
{
  "fixture_id": "...",
  "mocks": [
    {
      "tool_name": "read_note",
      "match": { "args": { "path": "..." } },
      "response": "..."
    }
  ],
  "on_unmocked_call": "fail_loudly"   // or "warn", "passthrough"
}
```

`on_unmocked_call: fail_loudly` is the default — any tool call from the model that isn't covered by a mock fails the fixture immediately. This catches the case where a strategy change causes the model to invoke an unexpected tool.

## Determinism

For numbers to be comparable across runs and across strategies, the runner pins:
- chat model + quant + temperature 0 + seed
- judge model + temperature 0 + three seeds
- embedding model + reranker
- workspace fixture (clean re-bootstrap per fixture run)
- `keep_alive: -1` for eval models
- random / time / UUID generation seeded per fixture

If you change any of those, you change the baseline.

## Adding a new fixture

The growth discipline is "add a fixture every time real usage produces a regression that the existing 10 didn't catch." Mirrors how [`backend/tests/eval/queries_reference.py`](../../queries_reference.py) grew from a small seed to 33 queries.

1. Pick a stable `id` (kebab-case). Increment the file-prefix number.
2. Author the conversation. Realistic content — domain-specific where relevant; the fixtures are most useful when they look like the customer's real usage.
3. Curate `expected_facts` carefully: every fact must be (a) actually present in the scripted history, (b) specific enough that a paraphrase-correct answer wouldn't accidentally fail, and (c) wrong enough to fail when the model confabulates.
4. Curate `must_not_contain` against the realistic confabulations — usually distractor data introduced into the conversation, or topic-bleed from earlier turns.
5. If the fixture uses tools, create the sibling `<id>.tools.json`. `on_unmocked_call: fail_loudly` is the default for new fixtures.
6. Run the fixture under `full-history` strategy locally to confirm a competent model passes mechanically.
7. Add an entry in this README's launch-fixture table.

## Status

Drafted 2026-04-27 alongside [ADR 010](../../../../../docs/architecture/decisions/010-conversation-replay-eval-harness.md). The runner that consumes these fixtures (`runner.py`, mocks, scorer, judge protocol) does not exist yet — these fixtures are the contract written first, code second. None of the fixtures have been replayed end-to-end against a real chat model; expected_facts patterns are author-curated and may need calibration once the runner lands and a `full-history` baseline is captured.
